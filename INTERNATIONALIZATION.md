# Internationalization Guide

## Overview

The extension uses VS Code's built-in l10n system for all user-visible strings. There are two distinct levels:

- **Package strings** (`package.nls.json`): command titles, view names, configuration descriptions, and other strings declared in `package.json` contribution points. VS Code reads these at load time.
- **Runtime strings** (`l10n/bundle.l10n.json`): messages, prompts, labels, and errors generated by TypeScript code during execution. These are resolved by `vscode.l10n.t()` (extension host) and `l10n.t()` from `@vscode/l10n` (webview).

The `"l10n"` field in `package.json` points VS Code to the `./l10n` directory, which is where bundle files are discovered automatically.

Currently supported locales: English (default) and Spanish (`es`).

## File Reference

| File                       | Purpose                                                                                                                             |
| -------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `package.nls.json`         | English strings for `package.json` (command titles, view names, configuration descriptions, walkthrough content, tool descriptions) |
| `package.nls.es.json`      | Spanish translations for `package.json`; keys mirror `package.nls.json`, values are translated strings                              |
| `l10n/bundle.l10n.json`    | English runtime strings with translator comments as values                                                                          |
| `l10n/bundle.l10n.es.json` | Spanish runtime strings; keys mirror `bundle.l10n.json`, values are translated strings                                              |

In the bundle files the convention differs from the package files: the key IS the English string, and the value in the English file is a translator comment (context note). In locale files the value is the translated string.

## Adding New Strings

### Strings in package.json contribution points

1. Add a descriptive key and its English value to `package.nls.json`:

   ```json
   "commands.myFeature.title": "Do Something Useful"
   ```

2. Add the same key with an empty string to `package.nls.es.json` (to be filled in by a translator):

   ```json
   "commands.myFeature.title": ""
   ```

3. Reference the key in `package.json` using the `%key%` syntax:

   ```json
   {
     "command": "datalayer.myFeature",
     "title": "%commands.myFeature.title%",
     "category": "%commands.category%"
   }
   ```

Key naming conventions used in this project:

| Prefix                                        | Used for                                |
| --------------------------------------------- | --------------------------------------- |
| `commands.<id>.title`                         | Command palette titles                  |
| `configuration.<setting>.description`         | Setting descriptions (plain text)       |
| `configuration.<setting>.markdownDescription` | Setting descriptions (Markdown)         |
| `views.<id>.name`                             | Tree view names                         |
| `walkthroughs.<id>.steps.<step>.*`            | Walkthrough step content                |
| `languageModelTools.<name>.*`                 | LLM tool display names and descriptions |

### Strings in extension TypeScript (src/)

The `vscode` namespace is available everywhere in the extension host; no extra import is needed.

1. Wrap the string with `vscode.l10n.t()`:

   ```typescript
   const label = vscode.l10n.t("Select a space to create the notebook in");
   ```

2. For strings with variables, use positional placeholders `{0}`, `{1}`, ...:

   ```typescript
   const msg = vscode.l10n.t('Connected to runtime "{0}"', runtime.givenName);
   ```

3. Add the English string as a key to `l10n/bundle.l10n.json` with a brief translator comment as the value:

   ```json
   "Select a space to create the notebook in": "Placeholder for space selection quick pick.",
   "Connected to runtime \"{0}\"": "Status message shown after connecting, {0} is the runtime display name."
   ```

4. Add the same key with an empty value to `l10n/bundle.l10n.es.json`:

   ```json
   "Select a space to create the notebook in": "",
   "Connected to runtime \"{0}\"": ""
   ```

5. Keep codicons and other non-translatable fragments outside the `l10n.t()` call:

   ```typescript
   // Correct - codicon is concatenated, not inside the translatable string
   const item = `$(cloud) ${vscode.l10n.t("Connect to Cloud Runtime")}`;

   // Wrong - do not include codicons inside l10n.t
   const item = vscode.l10n.t("$(cloud) Connect to Cloud Runtime");
   ```

### Strings in webview React (webview/)

The webview sandbox does not have access to the `vscode` namespace, so use the `@vscode/l10n` package instead.

1. Import the package:

   ```typescript
   import * as l10n from "@vscode/l10n";
   ```

2. Use `l10n.t()` exactly as you would use `vscode.l10n.t()`:

   ```typescript
   const placeholder = l10n.t("Enter notebook name");
   const error = l10n.t("Failed to create notebook: {0}", errorMessage);
   ```

3. Add strings to the same `l10n/bundle.l10n.json` and `l10n/bundle.l10n.es.json` files described above. There is a single bundle shared by both the extension host and webviews.

## Adding a New Language

1. Create `package.nls.<locale>.json` at the repository root. Copy all keys from `package.nls.json` and fill in the translated values:

   ```bash
   cp package.nls.json package.nls.fr.json
   # Edit package.nls.fr.json - fill in French translations
   ```

2. Create `l10n/bundle.l10n.<locale>.json`. Copy all keys from `l10n/bundle.l10n.json` and fill in the translated values (the English translator comments become the translated strings):

   ```bash
   cp l10n/bundle.l10n.json l10n/bundle.l10n.fr.json
   # Edit l10n/bundle.l10n.fr.json - replace comment values with French translations
   ```

3. That is all. VS Code discovers locale files automatically based on the user's display language. No code changes or registration are needed.

## String Conventions

### What to localize

Localize everything shown to users: labels, button text, prompts, progress messages, error messages displayed in notifications or quick picks, tooltip text, and placeholder text.

### What not to localize

Do not pass the following through `l10n.t()`:

- Log messages written to the output channel (developers read these in English)
- `Error` constructor arguments for internal errors that are never surfaced directly to users
- Command IDs (e.g., `datalayer.login`)
- API paths (e.g., `/api/spacer/v1/spaces`)
- Configuration setting IDs
- Proper nouns and brand names (GitHub, Pyodide, Datalayer, Jupyter)

### Common mistakes to avoid

- **Template literal messages**: Never use `` `Failed: ${error}` `` for user-facing strings. Always use `vscode.l10n.t("Failed: {0}", error)` so the entire sentence is translatable.
- **Partial localization**: Never mix localized fragments with English template literals (e.g., `` `Auth failed: ${l10n.t("Unknown error")}` ``). Either localize the whole string or keep it fully in English.
- **String concatenation for word order**: Never build UI text by concatenating `action + " " + item`. Use `l10n.t("{0} {1}", action, item)` so translators can reorder for their grammar.
- **Plural suffixes**: Never build plurals with a suffix parameter (`"runtime" + "s"`). Use separate singular and plural strings.
- **Button label comparison**: Store button labels in variables before passing to `showInformationMessage` and compare the return value against the same variable — never against a fresh `l10n.t()` call.

### Spanish translation quality

When writing Spanish translations, always verify:

- Accents on common words: sesión, más, dónde, debería, creación, configuración, código, extensión, información, conexión, aplicación, descripción, selección, acción, eliminación, confirmación, notificación, versión, automático, específico, número, público, página, índice, según, también, único, último, además, próximo, búsqueda, después
- Opening punctuation for questions and exclamations: use both marks
- Proper verb conjugation (hara, realizaran, etc. need accents)

### Key format (bundle files)

The English string is the key. Do not invent abstract key names:

```json
// Correct
"No spaces available": "Error when no Datalayer spaces exist."

// Wrong - abstract key in value position
"error.spaces.none": "No spaces available"
```

### Positional parameters

Use `{0}`, `{1}`, ... for runtime values. Mention each placeholder in the translator comment so translators understand what will be substituted:

```json
"Creating notebook \"{0}\" in space \"{1}\"...": "Progress title during notebook creation, {0} is the name, {1} is the space name."
```

## Scripts That Interact with i18n

### scripts/sync-package-json.ts

Syncs the `languageModelTools` contribution in `package.json` from TypeScript tool definitions. When tools are added, renamed, or removed, this script:

- Regenerates the `languageModelTools` array in `package.json` with `%key%` references
- Adds new `languageModelTools.<name>.displayName` and `languageModelTools.<name>.modelDescription` keys to `package.nls.json`
- Adds the same keys with empty values to `package.nls.es.json`
- Removes stale `languageModelTools.*` keys from both NLS files

Run it with:

```bash
npm run sync:tools
```

Do not manually edit `languageModelTools` entries in `package.json` or their corresponding NLS keys; they will be overwritten on the next sync.

### scripts/validate-pyodide-version.js

Reads the installed `pyodide` npm package version and syncs it into several TypeScript source files. It also updates the version string embedded in the `configuration.datalayer.pyodide.version.markdownDescription` key in `package.nls.json` so the configuration description always reflects the actual bundled version.

This script runs automatically before `compile` and `build`. It writes directly to `package.nls.json`; do not hardcode a Pyodide version in that file.

## Testing with a Different Locale

To verify translations during development, launch VS Code with the `--locale` flag:

```bash
code --locale es /path/to/workspace
```

Alternatively, set the display language inside VS Code:

1. Open the Command Palette (`Cmd+Shift+P` / `Ctrl+Shift+P`).
2. Run **Configure Display Language**.
3. Select `es` (or any installed locale).
4. Reload the window when prompted.

Empty values in the locale files (`""`) fall back to the English string automatically, so partial translations are safe to ship.
